Style

You’re starting to become something of C experts, and that means the code you write should reflect that. With that in mind, we’re going to discuss style. So far I’ve not really been picky about style (with the exception of annoying facets like soft tabs), but now that you’ve matured into fairly advanced C coding, it’s time to start coding efficiently and following C conventions.

  • Pick a capitalization style and be consistent. C is actually very friendly towards the underscore style (you’ll often see uint_32 type). Despite my leanings towards this style (this is C, after all,) I don’t particularly care whether you use camelcase or underscores so long as you’re consistent.
  • Code defensively – always error check.
  • Avoid using global variables; pass by reference instead to change values.
  • Line length is critical. No lines over 80 characters. C is used for kernel development, and this is a requirement. Comments, actual code – nothing can be over eighty characters in length.
  • Comment your code part 1: Use block comments. There should be block comments above each function that illustrate what the function does, what each of the parameters are, and what the return value is, if one exists.
  • Comment your code part 2: Use Inline comments. Inline comments are used for any particularly tricky or non-obvious lines of code. With C pointers, especially, there can be many strange things happening.
